Commercial model

Monthly cost in Indian rupees

Atlas Flex is pay-as-you-go and billed hourly. For 30 days of continuous use, the official price ranges from $8 to $30 per month. The amount rises with the cluster's operations-per-second usage band.

What can create additional cost?

  • The Flex database fee itself is capped at $30/month for a full month of usage.
  • Other Atlas services used separately—such as certain Search, Stream Processing, Data Federation, Charts or support capabilities—can appear as separate line items.
  • Indian tax, bank FX markup and currency movement are outside the USD list price.
Monthly estimate = official USD usage band × invoice-day USD/INR + applicable tax/FX charges
Capacity model

How much CRM data fits in 5 GB?

Flex counts dataSize + indexSize. It uses logical/uncompressed BSON size, so compression must not be used to claim extra capacity. The 5 GB ceiling cannot be expanded inside Flex.

Recommended operating envelope: use only 4 GB (80%); keep 1 GB as growth, index and incident headroom.

People & workload

How many users can the CRM support?

Three different “user” concepts: (1) CRM application users/employees stored as documents—potentially thousands; (2) simultaneously active users—limited by workload; (3) Atlas database users—credentials used by services/admins, with a project-level limit of 100. Do not create one database credential per employee.
Workload profileAssumption per active userMath at 100 ops/sRecommended simultaneous users
Light viewing~0.125 ops/s (one DB operation every 8 sec)≈800 theoretical200–400

Recommended bands apply conservative shared-cluster headroom and are not MongoDB guarantees. One API click can cause several database operations. Background jobs and CCP sync also consume operations.

500
Concurrent database connections

Use one pooled connection set per backend instance. A CRM employee should not open a direct Atlas connection.

500 ops/s
Hard throughput ceiling

Above the rate, Atlas queues/throttles work. The $8 band is based on up to 100 ops/s usage.

Recommended Node/Mongoose connection policy

  • Reuse a single process-level connection pool; never connect/disconnect for every API request.
  • Start with a modest pool (for example 10–20 per backend instance), then tune from metrics.
  • Limit serverless/backend instance fan-out so aggregate pools remain well below 500.
  • Debounce search, paginate list screens and cache slow-changing dashboard aggregates.
  • Run bulk imports in controlled batches outside peak hours.
Estimated DB ops/s = active users × operations per active user + sync/jobs/background traffic

BackupOne managed daily snapshotNo custom schedule, on-demand snapshot or point-in-time restore
MonitoringConnections, logical size, network, opscounter alerts/metricsNo Performance Advisor or full real-time panel

SecurityAtlas authentication, IP access list, TLS and managed platform securityNo customer-managed encryption key or database auditing on Flex

Operational restrictions worth knowing

  • Aggregation pipelines: maximum 50 stages.
  • In-memory sort limit: 32 MB.
  • No allowDiskUse for queries/aggregations.
  • Maximum BSON nesting depth: 50.
  • No configurable maintenance window.
  • No downloadable database logs.
  • No sharded deployment.
  • Flex resources are shared, not dedicated.
Business-critical warning: A daily snapshot can allow up to approximately one day of data loss between snapshots. If the CRM requires near-zero recovery point, point-in-time restore, private networking, audit logs or customer-managed keys, choose Dedicated M10+ rather than Flex.

Monthly operating routine

  1. Record dataSize, indexSize, peak connections, average/peak opscounter and network.
  2. Review top CRM endpoints by latency and database calls per request.
  3. Remove only confirmed unused indexes; do not optimize blindly.
  4. Forecast months-to-4.25 GB using the latest three months of growth.
  5. Load test before onboarding a large team, client import or new CCP integration.
Avoid false economy: If the cluster stays in the $21–$30 bands, queues during peak hours, or requires stronger recovery/security, optimizing indefinitely on Flex may cost more in operational risk than upgrading to Dedicated.
