Executive Summary

This CRM is a full-stack operational platform that joins lead acquisition, client onboarding, quotation, approval, compliance/annual-return processing, user/team administration, notifications and scheduled work. It uses a React single-page application, an Express service layer, MongoDB persistence and a protected integration boundary with CCP.

Login page

  • Accepts the user's registered email.
  • Calls POST /auth/request-otp.
  • Moves the user to verification after successful request.
  • Designed to avoid sharing passwords during normal login.

Verify OTP page

  • Verifies the time-bound code through /auth/verify-otp.
  • Supports OTP resend.
  • Stores authenticated user and token for API calls.
  • ProtectedRoute blocks unauthenticated business pages.

Security meaning

TermExplanation
JWTA signed session token sent as Authorization: Bearer …. Backend verifies it and loads the active user.
OTP expiryA verification code is useful only within a defined time window, reducing replay risk.
Active user checkEven a structurally valid token is rejected when its user is missing or inactive.
Role guardrequireRoles returns HTTP 403 when the authenticated role cannot perform an action.
Enterprise recommendation: production must set a strong JWT_SECRET; remove reliance on the development fallback, add refresh/session revocation, audit login events and enforce OTP throttling.

How CRM Is Linked to CCP

Implemented business flow: a Lead created in CCP is read into CRM and appears in Lead Generation and Dashboard. A Client available in CCP is read into CRM and appears in Client Master and Dashboard. CRM does not blindly duplicate the screen data; it normalizes CCP fields and merges them with CRM-owned records for one unified view.

Outbound synchronization: CRM → CCP

Trust boundary

CCP_SHARED_SECRET is transmitted as x-ccp-secret for protected machine-to-machine calls. Human browser calls still require JWT. Sync helpers return structured success/failure results so CRM actions can complete while reporting CCP sync status.

Enterprise recommendation: add an outbox queue with retry/backoff, idempotency keys, correlation IDs, signed requests, sync dashboards and dead-letter handling. Direct synchronous HTTP alone is vulnerable to temporary CCP downtime.
